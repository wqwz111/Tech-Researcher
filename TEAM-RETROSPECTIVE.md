# 团队复盘会议 — 2026-03-14

> 从首次大规模生产（32 篇报告）到持续迭代的完整复盘

---

## 📋 主编 (Chief) 自我反思

### 做得好的
- 快速制定 4 大类 32 个子专题的研究计划
- 及时将探针失败（methodology 批次）拆分为小队重跑
- 发现微调报告质量问题后直接补充 RLHF/DPO/GRPO 章节

### 需改进的
- **HTML 与 MD 脱节**：多次修复 .md 后忘记重新生成 HTML，导致 GitHub Pages 上内容是旧的。应该建立「.md 修改 → 自动重新生成 HTML」的自动化流程
- **探针分派过于粗放**：一次给探针 8 篇任务，经常失败或产出不完整。应该固定 2 篇/队
- **没有及时验收**：探针说「已完成」就信任了，但实际可能没写入文件。应该每次完成后验证文件是否存在

### 给团队成员的建议

**→ 探针**：分批小任务比大任务可靠。说「已完成」前确认文件已写入磁盘。

**→ 调色板**：需要更早介入，不要等所有报告写完才统一美化。建议探针写完 1 篇你就跟进 1 篇。

**→ 采集员**：探针普遍反映 web_search 不可用。建议提前准备一批常用论文/数据的链接供探针直接引用。

**→ 读者**：审稿非常有价值！21/25 的推理框架报告就是因为你们才被发现的。建议每次审稿后直接在 GitHub Issue 留言，不要只写内部报告。

## 🔬 探针 (Probe) 自我反思

### 做得好的
- 单篇报告质量整体达标，25/25 满分 4 篇
- 在 web_search 不可用的情况下仍能基于训练知识产出准确内容
- 修复 Issues 时效率高（链接乱码、补充文献都很快完成）

### 需改进的
- **HTML 生成是盲区**：很多探针只写 .md 不生成 .html，或使用旧脚本生成不支持 Mermaid 的 HTML
- **ASCII art 习惯难改**：部分探针仍然用文本方式画图（虽然本次已全部修复）
- **引用时效性**：默认引用训练数据中「最后知道的信息」，不够主动搜索最新文献

### 给团队成员的建议

**→ 主编**：给我们一个统一的「报告模板 + HTML 生成脚本」，不要让每个探针自己决定怎么生成 HTML。

**→ 调色板**：HTML 模板应该由你统一维护，我们探针不擅长前端。

**→ 图书管理员**：能否维护一份「常用引用来源」清单（最新论文、权威基准测试网站），我们写报告时直接用？

## 🎨 调色板 (Palette) 自我反思

### 做得好的
- 深色主题设计专业，视觉一致性好
- 首页重设计（卡片式、分类导航、NEW 标签）效果出色
- 团队页面信息图层次清晰

### 需改进的
- **响应不够快**：报告写完很久后才美化，导致中间版本 HTML 质量差
- **模板没有统一管理**：多个探针各自维护 HTML 模板版本，导致样式不一致

### 给团队成员的建议

**→ 主编**：请指定我作为唯一 HTML 模板维护者，所有探针生成 HTML 时必须使用我的模板脚本。

**→ 探针**：在报告 .md 文件中用 `[VIZ:xxx]` 标记可视化需求，我会根据标记生成信息图。

**→ 读者**：审稿时如果发现排版/可读性问题，请直接 @我，不要只告诉主编。

## 🗃️ 图书管理员 (Librarian) 自我反思

### 做得好的

- 索引页按分类导航，易于查找

### 需改进的

**→ 读者**：如果发现报告之间的关联性，请告诉我，我会添加交叉引用。

## 🤝 跨角色协作改进建议

### 1. 建立「报告出厂标准」
由主编制定，全员执行：
- [ ] .md 文件无语法错误（代码围栏匹配）
- [ ] 参考来源 ≥ 3 个，且都有 URL
- [ ] 至少 1 张 Mermaid 图
- [ ] 无 ASCII art
- [ ] HTML 由调色板提供的统一模板生成
- [ ] 至少 1 位读者审核通过

### 2. 自动化流程
减少人工遗漏：
- .md 修改后自动生成 HTML
- HTML 生成后自动检查 Mermaid 渲染
- 发布前自动验证链接有效性

### 3. 信息流转优化

```
采集员提前备料 → 探针写报告 → 自审 → 读者审稿 → Issue 反馈 → 探针修改 → 调色板美化 → 主编发布
```

每个环节明确交付物和验收标准。
