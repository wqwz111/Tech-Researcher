Executive Summary

OpenClaw 是一个开源的个人 AI 助手框架，它不是另一个聊天机器人 SaaS，而是一个跑在你自己设备上的 Gateway（网关）。它的核心理念是：你的设备、你的数据、你的 AI。通过统一的 Gateway 控制平面，OpenClaw 能将 20+ 通讯渠道（WhatsApp、Telegram、Slack、Discord、Signal、iMessage 等）与多个 AI Agent 统一编排，实现从"一个 AI 助手"到"一整支 AI 团队"的演进。本文深入剖析 OpenClaw 的核心架构组件、配置模型和路由机制，帮助技术决策者和开发者全面理解这个框架的设计哲学与实现细节。

1. OpenClaw 是什么？

OpenClaw 是一个开源的 AI Agent 基础设施框架，采用 MIT 许可证。与 ChatGPT、Claude 等云端 AI 服务不同，OpenClaw 的核心设计哲学是 "Local-first"——它跑在你自己的机器上（笔记本、台式机、树莓派、VPS 均可），你完全掌控数据流向和隐私边界。

🔑 核心定位：OpenClaw 不是一个 AI 模型，也不是一个聊天应用。它是一个 Agent Runtime（Agent 运行时）——连接你的通讯工具与 AI 模型之间的桥梁，负责消息路由、Agent 生命周期管理、工具执行和会话持久化。

1.1 为什么需要 OpenClaw？

在 OpenClaw 之前，如果你想让 AI 助手同时接入 WhatsApp 和 Telegram，你需要：分别写两个 bot、分别处理认证、分别管理状态、分别处理消息格式。OpenClaw 把这些"横切关注点"统一收编到一个 Gateway 中：

  • 统一入口：一个 Gateway 实例管理所有渠道
  • 统一配置：一个 openclaw.json 文件描述所有 Agent 和路由规则
  • 统一运行时：基于 Node.js，支持 npm/pnpm/bun
  • 数据自主：所有数据存储在本地，不经过第三方服务器

1.2 支持的通讯渠道

OpenClaw 内置了 20+ 通讯渠道的连接器，覆盖主流 IM 平台和协作工具：

类别渠道说明
即时通讯WhatsApp, Telegram, Signal, iMessage, SMS个人和群组消息
协作平台Slack, Discord, Microsoft Teams工作区/服务器内消息
开发工具GitHub (Issues/PR), GitLab, Linear代码协作自动化
Web 标准HTTP/WebSocket, Webhook自定义集成
其他Email (SMTP/IMAP), CLI传统渠道接入

每个渠道通过 Channel Provider 接入 Gateway，Channel Provider 负责将原生消息格式标准化为 OpenClaw 内部消息协议。

2. 核心组件架构

OpenClaw 的架构由四个核心组件构成，它们的关系可以理解为：

┌─────────────────────────────────────────────────┐
│                    Gateway                       │
│              (控制平面 / Control Plane)           │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────────┐   │
│  │ Binding  │  │ Router   │  │ Session Mgr  │   │
│  │ System   │→ │          │→ │              │   │
│  └──────────┘  └──────────┘  └──────────────┘   │
│       ↑             ↓               ↓            │
│  ┌──────────┐  ┌──────────┐  ┌──────────────┐   │
│  │ Channels │  │  Agent   │  │  Workspace   │   │
│  │ (20+)    │  │ Runtime  │  │  (per agent) │   │
│  └──────────┘  └──────────┘  └──────────────┘   │
└─────────────────────────────────────────────────┘

2.1 Gateway — 控制平面

Gateway 是 OpenClaw 的核心进程，也是唯一需要常驻运行的组件。它承担以下职责：

  • Channel 管理：监听所有已配置渠道的连接，处理认证、心跳、消息收发
  • 路由决策：根据 Binding 规则，决定每条入站消息应该交给哪个 Agent 处理
  • Agent 生命周期：按需启动/停止 Agent 实例，管理其进程和资源
  • Session 管理：维护每个 Agent 的会话状态、消息历史和上下文窗口
  • 工具注册：统一管理所有 Agent 可用的工具（Tools）和 MCP 服务器
💡 设计洞察：Gateway 本身不做 AI 推理。它是一个纯粹的"消息路由器 + 生命周期管理器"，所有 LLM 调用都在 Agent 层完成。这种关注点分离让 Gateway 保持轻量，也让 Agent 的替换和升级变得简单。

2.2 Agent — 大脑

Agent 是真正执行 AI 推理和工具调用的组件。每个 Agent 拥有完整的独立性：

  • 独立的 Workspace：文件系统沙箱，Agent 在其中读写文件、执行命令
  • 独立的 AgentDir：存放 Agent 的提示词（system prompt）、人设文件（SOUL.md）、指令文件等
  • 独立的 Session Store：持久化会话历史，支持跨重启恢复上下文
  • 独立的模型配置：每个 Agent 可以使用不同的 AI 模型（GPT-4o、Claude、Gemini、本地模型等）
  • 独立的工具集：通过 tools 配置精确控制每个 Agent 能使用哪些工具

2.3 Channel — 通讯渠道

Channel 是 OpenClaw 与外部世界的接口。每个 Channel Provider 实现以下功能：

  • 与目标平台的 API 建立连接（WebSocket、Long Polling、Webhook 等）
  • 将平台原生消息格式转换为 OpenClaw 统一消息协议
  • 将 Agent 的回复转换回平台原生格式发送
  • 处理平台特定的功能（如 Discord Embed、Telegram Inline Keyboard 等）

2.4 Workspace — 工作区

Workspace 是 Agent 的"工作目录"，类似于 Docker 容器的文件系统隔离概念：

  • 每个 Agent 有独立的 Workspace 路径
  • Agent 在 Workspace 中可以自由创建、读取、修改文件
  • Workspace 路径限制了 Agent 的文件系统可见范围（沙箱效应）
  • 可以配置为只读（workspaceReadOnly: true）

3. Agent 的完整定义

在 OpenClaw 中，一个完整的 Agent 定义由三个部分组成：

Agent = Workspace + AgentDir + Session Store

┌──────────────────────────────────┐
│              Agent               │
│                                  │
│  ┌────────────┐ ┌────────────┐  │
│  │ Workspace  │ │  AgentDir  │  │
│  │            │ │            │  │
│  │ • 项目文件  │ │ • SOUL.md  │  │
│  │ • 工作产出  │ │ • AGENTS.md│  │
│  │ • 临时数据  │ │ • USER.md  │  │
│  └────────────┘ │ • tools/   │  │
│                 └────────────┘  │
│  ┌────────────────────────┐     │
│  │     Session Store      │     │
│  │  • 消息历史             │     │
│  │  • 上下文窗口           │     │
│  │  • 会话元数据           │     │
│  └────────────────────────┘     │
└──────────────────────────────────┘

3.1 Workspace

Workspace 是 Agent 的工作文件系统。在这个目录下，Agent 可以：

  • 读写项目文件
  • 执行 shell 命令（在沙箱内）
  • 生成报告、代码、文档等产出物
  • 存储临时数据和缓存

3.2 AgentDir

AgentDir 是 Agent 的"人格定义"目录，包含以下关键文件：

文件作用示例内容
SOUL.md定义 Agent 的人设、语气、核心原则"你是一个冷静的技术分析师"
AGENTS.md定义 Agent 的工作流程和行为规则研究步骤、输出格式要求
USER.md描述服务对象的信息用户名、时区、偏好
TOOLS.md本地工具配置笔记SSH 主机、设备名称
MEMORY.md长期记忆，跨会话持久化重要发现、用户偏好
IDENTITY.mdAgent 的身份信息名字、emoji、头像

3.3 Session Store

Session Store 负责持久化 Agent 的会话状态。每个对话（或线程）对应一个 Session，包含：

  • 完整的消息历史记录
  • 当前上下文窗口（Context Window）
  • 工具调用记录和结果
  • 会话元数据（创建时间、最后活跃时间、关联渠道等）

4. 单 Agent vs 多 Agent 模式

OpenClaw 支持两种运行模式，从简单到复杂平滑过渡：

4.1 单 Agent 模式（默认）

最简单的配置——一个 Gateway + 一个 Agent。适合个人使用场景：

{
  "agents": {
    "defaults": {
      "model": "openrouter/anthropic/claude-sonnet-4"
    }
  },
  "channels": {
    "whatsapp": { "enabled": true },
    "telegram": { "enabled": true, "token": "..." }
  }
}

在单 Agent 模式下，所有渠道的消息都由同一个 Agent 处理。Gateway 会自动使用默认配置创建这个 Agent。

4.2 多 Agent 模式

通过 agents.list 定义多个 Agent，每个 Agent 有独立的配置、模型和行为：

{
  "agents": {
    "defaults": {
      "model": "openrouter/anthropic/claude-sonnet-4"
    },
    "list": [
      {
        "id": "daily",
        "model": "openrouter/google/gemini-2.5-flash",
        "workspace": "./workspaces/daily"
      },
      {
        "id": "deep-work",
        "model": "openrouter/anthropic/claude-opus-4",
        "workspace": "./workspaces/deep-work"
      },
      {
        "id": "family",
        "model": "openrouter/google/gemini-2.5-flash",
        "workspace": "./workspaces/family",
        "tools": { "allow": ["read", "web_search", "weather"] }
      }
    ]
  }
}
⚡ 何时升级到多 Agent：
  • 需要不同渠道用不同模型（省钱或性能优化）
  • 需要为不同人/群组提供不同能力的助手
  • 需要工具隔离（防止某些 Agent 执行危险操作）
  • 需要独立的 Workspace 避免文件交叉污染

5. 路由规则：Binding 系统

Binding 系统是 OpenClaw 消息路由的核心。它决定了"哪条消息应该由哪个 Agent 处理"。Binding 是确定性路由——不是基于 LLM 的智能判断，而是基于精确匹配规则。

5.1 Binding 的匹配维度

维度说明示例
channel按渠道类型匹配"whatsapp", "telegram"
account按账号 ID 匹配"whatsapp:+86138xxxx"
peer按对话方匹配"telegram:12345678"
guild按服务器/群组匹配"discord:987654321"
role按角色匹配"admin", "family"
thread按线程匹配"discord-thread:111"

5.2 Binding 配置示例

{
  "bindings": [
    {
      "agent": "daily",
      "match": { "channel": "whatsapp" }
    },
    {
      "agent": "deep-work",
      "match": { "channel": "telegram" }
    },
    {
      "agent": "family",
      "match": {
        "channel": "whatsapp",
        "peer": "whatsapp:+86138xxxx"
      }
    },
    {
      "agent": "coding",
      "match": {
        "channel": "discord",
        "guild": "discord:123456",
        "role": "admin"
      }
    }
  ]
}
🎯 路由优先级：Binding 规则按从上到下的顺序匹配，第一个匹配的规则胜出。因此，更具体的规则应该放在前面，通用规则放在后面。未匹配的消息会使用 default agent 处理。

6. 配置架构：openclaw.json

openclaw.json 是 OpenClaw 的唯一配置文件，采用 JSON 格式。以下是完整的配置结构：

{
  // Gateway 全局配置
  "gateway": {
    "port": 3000,
    "host": "0.0.0.0",
    "logLevel": "info"
  },

  // Agent 全局默认值
  "agents": {
    "defaults": {
      "model": "openrouter/anthropic/claude-sonnet-4",
      "maxContextTokens": 128000,
      "temperature": 0.7,
      "subagents": {
        "model": "openrouter/google/gemini-2.5-flash",
        "maxConcurrent": 5,
        "runTimeoutSeconds": 300
      }
    },
    "list": [
      // Agent 定义数组...
    ]
  },

  // 渠道配置
  "channels": {
    "whatsapp": { "enabled": true },
    "telegram": { "enabled": true, "token": "BOT_TOKEN" },
    "discord": { "enabled": true, "token": "BOT_TOKEN" },
    "slack": { "enabled": false },
    "signal": { "enabled": false }
  },

  // 路由绑定
  "bindings": [
    // Binding 规则数组...
  ],

  // MCP 服务器
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/data"]
    }
  },

  // 沙箱配置
  "sandbox": {
    "mode": "docker",
    "scope": "per-agent"
  }
}

8. 实际部署建议

8.1 个人使用（起步推荐）

# 1. 安装
npm install -g openclaw

# 2. 初始化配置
openclaw init

# 3. 配置渠道（以 Telegram 为例）
# 编辑 openclaw.json，添加 Telegram token

# 4. 启动
openclaw gateway start

# 5. 查看状态
openclaw status

8.2 VPS 部署

推荐使用 systemd 管理 OpenClaw 进程：

# /etc/systemd/system/openclaw.service
[Unit]
Description=OpenClaw Gateway
After=network.target

[Service]
Type=simple
User=openclaw
WorkingDirectory=/home/openclaw
ExecStart=/usr/bin/openclaw gateway start --foreground
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target

9. 总结

OpenClaw 的架构设计体现了几个关键理念：

  1. 关注点分离：Gateway 只做路由和生命周期管理，Agent 只做推理和工具调用
  2. 确定性路由：Binding 系统用精确规则而非 AI 判断来路由消息，保证可预测性
  3. 独立沙箱：每个 Agent 的 Workspace、模型、工具集完全隔离
  4. 配置驱动：一个 JSON 文件描述整个系统，版本可控、易于复制
  5. Local-first：数据在你自己的设备上，完全掌控隐私和安全

对于想从"用一个 AI 聊天"升级到"管理一支 AI 团队"的用户，OpenClaw 提供了一个坚实的技术基础。在后续报告中，我们将深入探讨多 Agent 团队协作、ACP 编程智能体集成和企业级部署模式。
