RAG 生产环境踩坑实录

📅 2025年3月📂 案例实践📖 ~4500字

📋 Executive Summary

RAG 已成为企业部署 LLM 最主流的架构模式。但从 PoC 到生产的过程中面临着大量意料之外的挑战。

  • 数据质量决定上限：再好的检索策略也无法弥补脏数据
  • Chunking 策略影响巨大：固定长度分块是最常见的反模式
  • 检索失败是最被低估的问题："找不到"比"找错"更常见
  • 幻觉无法完全消除：但可通过工程手段控制

1. 真实环境中的 RAG 挑战

PoC vs Production 的巨大鸿沟：文档从"干净 PDF"变成各种格式混杂，用户从"内部测试"变成"千奇百怪的问题"。

⚠️ 典型陷阱：检索到目录页而非正文、检索到过时版本、多文档综合问题失败、无文档支持但 LLM 仍"编造"回答。

2. 数据质量与清洗

清洗管线：格式检测 → 文本提取 → OCR（扫描件）→ 清洗 → 分块 → 嵌入 → 存储。

元数据是被低估的部分：文档标题、作者、日期、版本号、文档类型、适用范围——这些是检索过滤和回答溯源的基础。

3. 检索失败模式与应对

混合检索策略

向量检索（语义）+ BM25（关键词）+ RRF 融合 + Reranker 精排，通常可将检索准确率提升 10-20%。

查询改写

问题分解、查询扩展、HyDE（用假设性答案检索）。

4. 幻觉控制策略

策略效果复杂度
Prompt 工程中低
引用溯源高中
置信度评分中中
事实一致性检查高高
安全边界规则高低

5. 监控与持续优化

四个监控维度：系统（延迟/错误率）、检索（相关性/失败率）、生成（幻觉率/Token 消耗）、业务（满意度/解决率）。

生产环境踩坑清单（Top 10）

  1. 嵌入模型更新后未重建索引
  2. 文档删除后未清理向量
  3. 并发突增导致向量数据库 OOM
  4. 长文档分块不当，关键信息丢失
  5. 多语言检索问题
  6. 时间敏感信息处理
  7. PDF 表格提取失败
  8. Prompt 注入攻击
  9. 成本失控
  10. 敏感信息未脱敏

实践建议

技术选型：向量数据库选 Pinecone/Qdrant/Milvus；嵌入模型选 OpenAI/BGE-M3；先 LangChain 原型再自建。

架构：从简单开始，先保证召回再优化精度，元数据过滤是低垂果实。

运营：每周抽样审查、建立文档更新流程、用户反馈 48 小时闭环。
