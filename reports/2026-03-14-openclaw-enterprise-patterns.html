Executive Summary

当 OpenClaw 从个人使用扩展到团队共享时，安全隔离、成本控制和运维监控成为关键挑战。本文系统性地介绍 OpenClaw 的企业级部署模式：通过 Docker 沙箱实现 Agent 间容器级隔离、通过 per-agent tools 白名单实现权限分级、通过 Sub-agent 模型分层实现成本优化、通过并发控制和会话管理实现资源管控。无论你是 3 人小团队还是 30 人的工程部门，这些模式都能帮你安全、高效地共享一个 Gateway 实例。

1. 多人共享 Gateway 的隔离挑战

多人共享一个 OpenClaw Gateway 实例时，面临的核心挑战是隔离：

隔离维度风险解决方案
数据隔离A 的文件被 B 的 Agent 读取per-agent Workspace + Docker 沙箱
工具隔离A 的 Agent 被滥用执行危险命令per-agent tools allow/deny 白名单
成本隔离A 用便宜模型，B 用贵模型，账单混在一起per-agent model 配置
进程隔离A 的 Agent 崩溃影响 B 的 AgentDocker 容器级进程隔离
会话隔离A 的对话历史泄露给 Bper-agent Session Store

2. 沙箱配置详解

2.1 沙箱模式

OpenClaw 支持两种沙箱模式：

模式隔离级别性能开销适用场景
none无隔离无完全信任的个人使用
docker容器级低（秒级启动）团队共享、不受信任的输入

2.2 Docker 沙箱配置

{
  "sandbox": {
    "mode": "docker",
    "scope": "per-agent",
    "docker": {
      "image": "openclaw/sandbox:latest",
      "setupCommand": "apt-get update && apt-get install -y git curl",
      "memoryLimit": "512m",
      "cpuLimit": "1.0",
      "networkMode": "bridge",
      "readonly": false
    }
  }
}
参数说明推荐值
mode沙箱模式"docker"
scope沙箱范围"per-agent"（每个 Agent 独立容器）
imageDocker 镜像根据需求选择基础镜像
setupCommand容器初始化命令安装 Agent 需要的工具
memoryLimit内存限制根据 Agent 负载调整
cpuLimitCPU 限制根据 Agent 负载调整
networkMode网络模式"bridge"（隔离网络）或 "host"（共享网络）
readonly只读文件系统高安全场景设为 true

2.3 Per-Agent 沙箱覆盖

全局沙箱配置可以用 per-agent 配置覆盖：

{
  "agents": {
    "list": [
      {
        "id": "trusted-dev",
        "model": "openrouter/anthropic/claude-sonnet-4",
        "sandbox": {
          "mode": "none"
        }
      },
      {
        "id": "untrusted-guest",
        "model": "openrouter/google/gemini-2.5-flash",
        "sandbox": {
          "mode": "docker",
          "docker": {
            "memoryLimit": "256m",
            "cpuLimit": "0.5",
            "readonly": true,
            "networkMode": "none"
          }
        }
      }
    ]
  }
}
🔒 安全原则：对于不受信任的输入来源（如外部群组、公开频道），始终使用 Docker 沙箱 + readonly + 无网络。即使被 prompt injection，攻击者也无法读取敏感文件或发起网络请求。

3. 工具权限分级

3.1 Per-Agent Tools 白名单

每个 Agent 可以独立配置工具白名单（allow）和黑名单（deny）：

{
  "agents": {
    "list": [
      {
        "id": "admin",
        "model": "openrouter/anthropic/claude-opus-4",
        "tools": {
          "allow": ["read", "write", "edit", "exec", "process", "web_search"]
        }
      },
      {
        "id": "developer",
        "model": "openrouter/anthropic/claude-sonnet-4",
        "tools": {
          "allow": ["read", "write", "edit", "exec", "web_search"],
          "deny": ["process"]
        }
      },
      {
        "id": "analyst",
        "model": "openrouter/google/gemini-2.5-flash",
        "tools": {
          "allow": ["read", "web_search"]
        }
      },
      {
        "id": "guest",
        "model": "openrouter/google/gemini-2.5-flash",
        "tools": {
          "allow": ["web_search", "weather"]
        }
      }
    ]
  }
}

3.2 工具权限矩阵

工具风险级别功能建议
read🟡 低读取文件大多数 Agent 可用
web_search🟡 低搜索网络大多数 Agent 可用
weather🟢 极低查询天气所有 Agent 可用
write🟠 中创建/覆盖文件仅信任 Agent
edit🟠 中修改文件仅信任 Agent
exec🔴 高执行 Shell 命令仅限管理员/开发 Agent
process🔴 高管理后台进程仅限管理员 Agent

4. 安全边界设计

4.1 三层防御模型

┌──────────────────────────────────────────────┐
│  Layer 1: Gateway 层安全                      │
│  • Binding 路由 → 消息只能到达授权的 Agent     │
│  • Channel 认证 → 只有授权用户能发消息         │
├──────────────────────────────────────────────┤
│  Layer 2: Agent 层安全                        │
│  • per-agent tools 白名单 → 工具级权限控制     │
│  • per-agent model → 模型级成本控制            │
│  • per-agent AgentDir → 人设隔离              │
├──────────────────────────────────────────────┤
│  Layer 3: 沙箱层安全                           │
│  • Docker 容器 → 进程级隔离                   │
│  • readonly FS → 文件系统保护                 │
│  • 网络隔离 → 阻止外部连接                    │
│  • 资源限制 → CPU/Memory 上限                │
└──────────────────────────────────────────────┘

4.2 受信任 vs 不受信任场景

场景信任级别推荐配置
个人 DM✅ 完全信任完整工具集，无沙箱
团队频道🟡 部分信任受限工具集，Docker 沙箱
公开群组🔴 不信任只读工具，Docker + readonly + 无网络
外部 Webhook🔴 不信任严格输入验证 + 最小权限 Agent

7. 成本优化策略

7.1 模型分层策略

最有效的成本优化手段是模型分层：不同任务用不同能力/价格的模型。

{
  "agents": {
    "defaults": {
      "model": "openrouter/anthropic/claude-sonnet-4",
      "subagents": {
        "model": "openrouter/google/gemini-2.5-flash"
      }
    },
    "list": [
      {
        "id": "executive",
        "model": "openrouter/anthropic/claude-opus-4"
      },
      {
        "id": "daily",
        "model": "openrouter/google/gemini-2.5-flash"
      }
    ]
  }
}

7.2 成本对比

模型Input $/1MOutput $/1M适合
Claude Opus 4$15$75复杂推理、关键决策
Claude Sonnet 4$3$15日常专业任务
GPT-4o$2.5$10通用任务
Gemini 2.5 Flash$0.15$0.60Sub-agent、日常问答
Gemini 2.5 Pro$1.25$5中等复杂度任务
💰 成本优化公式：让 90% 的消息由便宜模型处理（Gemini Flash），10% 的关键任务由强大模型处理（Claude Opus）。通过 Sub-agent 分流，主 Agent 用强模型做决策，Sub-agent 用便宜模型做执行。理论上可降低 70-80% 的 AI 调用成本。

7.3 Sub-agent 成本控制

{
  "agents": {
    "defaults": {
      "model": "openrouter/anthropic/claude-sonnet-4",
      "subagents": {
        "model": "openrouter/google/gemini-2.5-flash",
        "maxConcurrent": 5,
        "runTimeoutSeconds": 300,
        "maxSpawnDepth": 1
      }
    }
  }
}

runTimeoutSeconds 防止 Sub-agent 无限运行消耗预算。300 秒（5 分钟）是一个合理的上限——超过这个时间的任务应该拆分为更小的子任务。

8. 并发控制

8.1 maxConcurrent

限制同时运行的 Sub-agent 数量，防止资源耗尽：

{
  "agents": {
    "defaults": {
      "subagents": {
        "maxConcurrent": 5
      }
    }
  }
}

8.2 maxChildrenPerAgent

限制单个 Agent 能同时拥有的子 Agent 数量：

{
  "agents": {
    "defaults": {
      "maxChildrenPerAgent": 3
    }
  }
}

8.3 并发控制策略建议

场景maxConcurrentmaxChildrenPerAgent理由
个人使用33资源有限，少量并发
小团队 (3-5人)53平衡响应速度和资源
大团队 (10+人)105高并发需求
资源受限 (512MB)22内存有限

9. 会话管理

9.1 自动归档

长时间不活跃的会话自动归档，释放内存和存储：

{
  "agents": {
    "defaults": {
      "archiveAfterMinutes": 1440,
      "threadIdleMinutes": 60,
      "threadMaxLifetimeMinutes": 10080
    }
  }
}
参数说明推荐值
archiveAfterMinutes会话无活动多久后自动归档1440（24小时）
threadIdleMinutes线程空闲多久后关闭60（1小时）
threadMaxLifetimeMinutes线程最大存活时间10080（7天）

9.2 会话生命周期

活跃 → 空闲(threadIdleMinutes) → 待归档 → 归档(archiveAfterMinutes)
                                           ↓
                                      可恢复(recoverable)
                                           ↓
                                      清理(cleanup)

10. 监控与诊断

10.1 openclaw status

查看 Gateway 整体状态：

$ openclaw status

Agents: 4 active, 2 idle
Channels: whatsapp(connected), telegram(connected), discord(connected)
Sessions: 12 active, 45 archived
Memory: 512MB / 2GB (25%)
Sub-agents: 3 running, 2 queued

10.2 /acp doctor

在聊天中诊断 ACP 环境：

/acp doctor

✅ Codex: v1.2.3, API key OK, sandbox OK
✅ Claude Code: v2.0.1, API key OK
⚠️ Gemini CLI: v0.8.0, API key OK, rate limited (retry in 30s)

10.3 日志分析

OpenClaw 的日志提供以下关键信息：

  • 消息路由决策（哪条消息路由到哪个 Agent）
  • Agent 启动/停止事件
  • 工具调用记录（含耗时）
  • 模型 API 调用记录（含 token 用量）
  • 错误和异常
# 查看实时日志
openclaw gateway logs --follow

# 按 Agent 过滤
openclaw gateway logs --agent coding --follow

# 查看错误日志
openclaw gateway logs --level error --since 1h

10.4 关键监控指标

指标正常范围告警阈值含义
Gateway 内存使用< 50%> 80%内存不足风险
活跃会话数< maxConcurrent> maxConcurrent * 0.8并发上限接近
模型 API 延迟< 5s> 30sAPI 响应慢
错误率< 1%> 5%系统异常
Docker 容器数< CPU 核数> CPU 核数 * 2容器过多

11. 完整生产配置示例

以下是一个面向 5 人团队的完整生产配置：

{
  "gateway": {
    "port": 3000,
    "host": "0.0.0.0",
    "logLevel": "info"
  },

  "agents": {
    "defaults": {
      "model": "openrouter/anthropic/claude-sonnet-4",
      "maxContextTokens": 128000,
      "archiveAfterMinutes": 1440,
      "threadIdleMinutes": 60,
      "threadMaxLifetimeMinutes": 10080,
      "subagents": {
        "model": "openrouter/google/gemini-2.5-flash",
        "maxConcurrent": 5,
        "runTimeoutSeconds": 300,
        "maxSpawnDepth": 1
      }
    },
    "list": [
      {
        "id": "admin",
        "model": "openrouter/anthropic/claude-opus-4",
        "workspace": "./workspaces/admin",
        "agentDir": "./agents/admin",
        "tools": {
          "allow": ["read", "write", "edit", "exec", "process", "web_search"]
        }
      },
      {
        "id": "dev",
        "model": "openrouter/anthropic/claude-sonnet-4",
        "workspace": "./workspaces/dev",
        "agentDir": "./agents/dev",
        "tools": {
          "allow": ["read", "write", "edit", "exec", "web_search"]
        },
        "sandbox": {
          "mode": "docker",
          "docker": { "memoryLimit": "512m", "cpuLimit": "1.0" }
        }
      },
      {
        "id": "support",
        "model": "openrouter/google/gemini-2.5-flash",
        "workspace": "./workspaces/support",
        "agentDir": "./agents/support",
        "tools": {
          "allow": ["read", "web_search"]
        },
        "sandbox": {
          "mode": "docker",
          "docker": {
            "memoryLimit": "256m",
            "cpuLimit": "0.5",
            "readonly": true
          }
        }
      }
    ]
  },

  "bindings": [
    {
      "agent": "admin",
      "match": { "role": "admin" }
    },
    {
      "agent": "dev",
      "match": { "channel": "discord", "guild": "discord:dev-server" }
    },
    {
      "agent": "support",
      "match": { "channel": "whatsapp", "group": "whatsapp:support-group" }
    }
  ],

  "channels": {
    "whatsapp": { "enabled": true },
    "telegram": { "enabled": true, "token": "${TELEGRAM_TOKEN}" },
    "discord": { "enabled": true, "token": "${DISCORD_TOKEN}" }
  },

  "sandbox": {
    "mode": "docker",
    "scope": "per-agent"
  }
}

12. 部署检查清单

  1. ✅ Gateway 使用 systemd 管理，配置自动重启
  2. ✅ 每个 Agent 有独立的 Workspace 路径
  3. ✅ 不受信任的 Agent 配置 Docker 沙箱 + readonly
  4. ✅ 工具白名单配置完成，高风险工具仅限管理员
  5. ✅ Sub-agent 使用便宜模型，设置 runTimeoutSeconds
  6. ✅ 并发限制合理设置，不超过服务器承载能力
  7. ✅ 会话自动归档配置完成
  8. ✅ 日志级别设置为 info，错误日志单独收集
  9. ✅ API Key 使用环境变量，不硬编码在配置文件中
