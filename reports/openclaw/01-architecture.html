📋 Executive Summary

OpenClaw 是一个开源的个人 AI 助手框架，让你在自有设备上运行 AI 助手，并通过 20+ 通讯渠道（WhatsApp, Telegram, Slack, Discord, Signal, iMessage 等）与其交互。本报告深入解析其四大核心组件——Gateway、Agent、Channel、Workspace，以及 Binding 路由系统的工作原理。理解这些架构概念是后续掌握多智能体团队管理、ACP 编程集成和企业级部署的基础。

🎯 OpenClaw 是什么？

OpenClaw 是一个 self-hosted（自托管）的个人 AI 助手。与 ChatGPT/Claude 等云端服务不同，OpenClaw 运行在你自己的设备上（笔记本、树莓派、VPS），数据完全掌控在你手中。

核心特性

  • 全渠道接入：20+ 通讯渠道统一到一个 AI 助手
  • 多模型支持：OpenAI, Anthropic, Google, 本地模型等
  • 多 Agent 隔离：一个 Gateway 可运行多个独立的 AI "大脑"
  • 子智能体编排：并行后台任务 + 嵌套编排器模式
  • ACP 编程集成：直接调度 Codex, Claude Code, Gemini CLI
  • 安全沙箱：Docker 容器隔离 + 工具权限分级

1. Gateway — 控制平面 ▾

Gateway 是整个系统的"神经中枢"。它不产生智能，而是负责：

  • 消息路由：收到消息后，根据 Binding 规则路由到对应的 Agent
  • 渠道管理：维护所有通讯渠道的连接（WebSocket, Polling, Webhook）
  • 会话管理：创建、存储、归档 Agent 会话
  • 进程守护：作为 systemd/launchd 服务持续运行
# Gateway 启动
openclaw gateway start --port 18789

# 查看状态
openclaw gateway status
openclaw status
2. Agent — AI 大脑 ▾

一个 Agent 是一个完整的、隔离的 AI 实体，由三部分组成：

组件路径作用
Workspace~/.openclaw/workspace-/文件、配置（AGENTS.md/SOUL.md/USER.md）、技能
AgentDir~/.openclaw/agents//agent/认证配置、模型注册表、per-agent 设置
Session Store~/.openclaw/agents//sessions/聊天历史、路由状态
💡 关键理解：每个 Agent 的认证（auth-profiles.json）是独立的。主 Agent 的凭证不会自动共享给其他 Agent。如果你想共享，需要手动复制 auth-profiles.json。
3. Channel — 通讯渠道 ▾

OpenClaw 支持 20+ 通讯渠道。每个渠道可以配置多个账号（account），每个账号对应一个独立的登录实例：

# 渠道登录示例
openclaw channels login --channel whatsapp --account personal
openclaw channels login --channel whatsapp --account biz
openclaw channels login --channel telegram --account default
4. Workspace — 工作区 ▾

Workspace 是 Agent 的"家"，包含定义其行为和个性的所有文件：

  • AGENTS.md — Agent 行为指南、工作流、规则
  • SOUL.md — 人设、语气、性格特征
  • USER.md — 用户信息、偏好
  • TOOLS.md — 环境特定配置（设备、SSH 等）
  • skills/ — Agent 专属技能

🔀 Binding 路由系统

Binding 是 OpenClaw 的"交通指挥系统"。它决定了哪条消息进入哪个 Agent。路由是确定性的（deterministic），遵循 most-specific wins 原则。

路由优先级（从高到低）

  1. peer 匹配（精确的 DM/群组/频道 ID）
  2. parentPeer 匹配（线程继承）
  3. guildId + roles（Discord 角色路由）
  4. guildId（Discord 服务器）
  5. teamId（Slack 工作区）
  6. accountId 匹配（渠道账号）
  7. 渠道级匹配（accountId: "*"）
  8. 回退到默认 Agent

Binding 配置示例

{
  bindings: [
    // 精确路由：特定 DM → Opus Agent
    {
      agentId: "opus",
      match: { channel: "whatsapp", peer: { kind: "direct", id: "+15551234567" } }
    },
    // 渠道路由：整个 Telegram → Opus Agent
    { agentId: "opus", match: { channel: "telegram" } },
    // 回退：其他所有 WhatsApp → chat Agent
    { agentId: "chat", match: { channel: "whatsapp" } },
  ]
}

Binding 冲突与边界场景

当多条 Binding 规则同时匹配时，OpenClaw 遵循 most-specific wins 原则。以下是常见边界场景：

场景 1：peer 与 accountId 同时匹配
规则 A：{ agentId: "opus", match: { peer: "+15551234567" } }
规则 B：{ agentId: "chat", match: { accountId: "personal" } }
结果：peer 优先级高于 accountId → 路由到 opus
场景 2：同一 peer 多条规则
规则 A：{ agentId: "deep", match: { peer: "+15551234567" } }
规则 B：{ agentId: "opus", match: { peer: "+15551234567" } }
结果：列表中靠前的规则生效 → 路由到 deep（配置顺序决定）
场景 3：无匹配规则
消息来源的 channel/peer/accountId 均未命中任何 Binding。
结果：回退到 agents.list 中的第一个 Agent（通常为 main）
建议：显式配置回退规则 { agentId: "chat", match: { channel: "whatsapp" } } 避免意外路由

🛠️ 技术栈

  • 运行时：Node.js ≥ 22
  • 包管理：npm, pnpm, bun 均支持
  • 守护进程：systemd (Linux) / launchd (macOS)
  • 配置语言：JSON5（支持注释和尾逗号）
  • 协议：各渠道原生协议 + ACP (Agent Client Protocol)
# 安装
npm install -g openclaw@latest

# 向导式配置（推荐）
openclaw onboard --install-daemon

# 手动启动
openclaw gateway --port 18789 --verbose
