📋 Executive Summary

OpenClaw 的多 Agent 架构让你在一个 Gateway 中运行多个完全隔离的 AI "大脑"，每个 Agent 拥有独立的 Workspace、认证和会话。通过 Binding 系统，你可以按渠道、按对话、甚至按群组角色精确路由消息。Sub-agent 系统支持并行后台任务和嵌套编排器模式（maxSpawnDepth=2），实现复杂的多智能体协作流水线。本报告通过 4 个真实场景案例，手把手教你组建和管理 AI Agent 团队。

🔀 Binding 六维路由

Binding 支持 6 个匹配维度，最具体的规则优先：

优先级匹配维度示例典型用途
1peer特定 DM 号码 / 群组 IDVIP 客户专用 Agent
2parentPeer线程继承子线程跟随父级路由
3guildId + rolesDiscord 服务器 + 角色管理员走 Opus，普通用户走 Sonnet
4guildIdDiscord 服务器整个服务器统一路由
5accountIdWhatsApp 号码工作号 vs 个人号
6渠道级channel: "telegram"整个渠道统一

📱 场景 A：按渠道分流

最经典的用法：WhatsApp 走快速模型做日常对话，Telegram 走高级模型做深度工作。

{
  agents: {
    list: [
      { id: "chat", model: "anthropic/claude-sonnet-4-5", workspace: "~/.openclaw/workspace-chat" },
      { id: "deep", model: "anthropic/claude-opus-4-6", workspace: "~/.openclaw/workspace-deep" },
    ]
  },
  bindings: [
    { agentId: "chat", match: { channel: "whatsapp" } },
    { agentId: "deep", match: { channel: "telegram" } },
  ]
}

💬 场景 B：同一渠道内按对话分流

保持 WhatsApp 作为主要渠道，但把特定 DM 或群组路由到专用 Agent：

bindings: [
  // 优先：老板的 DM 走 Opus
  {
    agentId: "deep",
    match: { channel: "whatsapp", peer: { kind: "direct", id: "+8613800138000" } }
  },
  // 优先：工作群走工作 Agent
  {
    agentId: "work",
    match: { channel: "whatsapp", peer: { kind: "group", id: "120363xxxx@g.us" } }
  },
  // 回退：其他所有 WhatsApp 走日常 Agent
  { agentId: "chat", match: { channel: "whatsapp" } },
]
💡 关键点：peer 绑定必须放在渠道级绑定之前！peer 绑定优先级更高，放在前面可读性更好。

🔄 Sub-agent 系统

Sub-agent 是从主 Agent 派生的后台任务。它在隔离会话中运行，完成后自动 announce 结果回主 Agent。

sequenceDiagram participant User participant Main as [ ] 主 Agent participant Sub1 as [ ] Sub-agent 1 participant Sub2 as [ ] Sub-agent 2 User->>Main: "分析这三个竞品" Main->>Sub1: sessions_spawn("分析竞品 A") Main->>Sub2: sessions_spawn("分析竞品 B") Note over Main: 主 Agent 继续其他工作 Sub1-->>Main: [ ] announce: 竞品 A 分析报告 Sub2-->>Main: [ ] announce: 竞品 B 分析报告 Main->>User: 汇总所有结果，输出完整报告

Sub-agent 配置

{
  agents: {
    defaults: {
      subagents: {
        model: "anthropic/claude-sonnet-4-5",  // 子智能体用便宜模型
        maxConcurrent: 8,                        // 全局并发上限
        maxChildrenPerAgent: 5,                  // 每个 Agent 最大子任务数
        runTimeoutSeconds: 900,                  // 15 分钟超时
        archiveAfterMinutes: 60,                 // 完成后 60 分钟归档
      }
    }
  }
}

🎼 嵌套 Sub-agent 编排器模式

设置 maxSpawnDepth: 2 后，Sub-agent 可以再派生子任务，形成三层编排架构：

{
  agents: {
    defaults: {
      subagents: {
        maxSpawnDepth: 2,     // 允许嵌套：编排器 → worker
        maxChildrenPerAgent: 5 // 每层最多 5 个子任务
      }
    }
  }
}

工具策略按深度分层

深度角色可用工具可再 spawn?
0主 Agent全部✅ 总是可以
1编排器sessions_spawn, subagents, sessions_list, sessions_history✅ 仅 maxSpawnDepth≥2
2Worker读写执行等（无 session 工具）❌ 永远不能

级联停止

停止编排器会自动停止所有子级 Worker：

  • /stop — 停止主 Agent → 级联停止所有 depth-1 和 depth-2
  • /subagents kill <id> — 停止指定 Sub-agent 及其子级
  • /subagents kill all — 停止所有 Sub-agent 及其子级
