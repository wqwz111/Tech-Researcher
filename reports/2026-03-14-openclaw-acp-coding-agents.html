Executive Summary

ACP（Agent Client Protocol）是 OpenClaw 用于集成外部编程智能体的协议层。通过 ACP，你可以从 WhatsApp、Telegram、Discord 等任何 OpenClaw 支持的渠道直接 spawn 和管理 Codex、Claude Code、Gemini CLI 等专业编程 Agent——不需要切换到终端或 IDE。本文深入剖析 ACP 的架构设计、两种运行模式、线程绑定机制和完整操作命令，并与原生 Sub-agent 系统进行对比分析。

1. ACP 是什么？

ACP（Agent Client Protocol）是一个开放协议，定义了宿主应用（Host Application）与编程 Agent（Coding Agent）之间的标准通信方式。在 OpenClaw 的上下文中：

  • 宿主应用 = OpenClaw Gateway（作为 ACP Client）
  • 编程 Agent = Codex、Claude Code、OpenCode、Gemini CLI 等（作为 ACP Server）
🔑 核心价值：ACP 让编程 Agent 变成 OpenClaw 生态中的"一等公民"。你可以像使用 Sub-agent 一样 spawn 编程 Agent，但它们运行在自己的独立运行时中（如 Codex 的沙箱容器），拥有完整的代码执行环境。

3. 两种运行模式

3.1 Run 模式（一次性）

Run 模式下，编程 Agent 执行一次性任务，完成后自动退出：

/acp spawn codex "Fix the failing tests in auth module"

流程：

  1. Gateway 创建一个临时 ACP 会话
  2. 将任务描述发送给编程 Agent
  3. 编程 Agent 执行任务（修改代码、运行测试等）
  4. 任务完成，编程 Agent 退出
  5. 结果通过 OpenClaw 回复用户

3.2 Session 模式（持久化）

Session 模式下，编程 Agent 保持运行，可以持续交互：

/acp spawn claude-code --mode session "Help me refactor the payment module"

流程：

  1. Gateway 创建一个持久 ACP 会话
  2. 编程 Agent 启动并等待输入
  3. 用户可以持续发送指令：/acp steer "also update the tests"
  4. 编程 Agent 在同一会话上下文中持续工作
  5. 用户主动关闭：/acp close
💡 选择建议：
  • Run 模式适合：明确的、范围有限的任务（修 bug、加单个功能、代码审查）
  • Session 模式适合：需要多轮交互的复杂任务（大型重构、架构改造、调试棘手问题）

4. 线程绑定机制

ACP 最强大的特性之一是线程绑定：在 Discord 或 Telegram 中，用户可以创建一个线程（Thread），该线程会自动路由到同一个 ACP 会话。

4.1 工作原理

Discord 服务器
  └── #coding 频道
        └── 消息: "@bot /acp spawn codex 'add login feature'"
              → Gateway 在 #coding 创建一个 Discord Thread
              → Thread 自动绑定到一个新的 ACP Session
              → 后续在 Thread 中的所有消息都路由到同一个 ACP Session
              
        └── Thread: "add login feature"
              ├── 消息 1: "add OAuth2 support"
              │   → /acp steer (路由到绑定的 ACP Session)
              ├── 消息 2: "also add unit tests"
              │   → /acp steer (路由到同一个 ACP Session)
              └── 消息 3: "/acp close"
                  → 关闭 ACP Session，Thread 归档

这种机制的优势：

  • 上下文连续性：编程 Agent 在同一个会话中积累上下文
  • 多任务并行：不同 Thread 绑定不同 ACP Session，互不干扰
  • 自然交互：用户像在群里聊天一样控制编程 Agent

5. 配置示例

5.1 Per-Agent Runtime 配置

{
  "acp": {
    "agents": {
      "codex": {
        "command": "codex",
        "args": ["acp"],
        "env": {
          "OPENAI_API_KEY": "${OPENAI_API_KEY}"
        },
        "permissionMode": "approve-all"
      },
      "claude-code": {
        "command": "claude",
        "args": ["--acp"],
        "env": {
          "ANTHROPIC_API_KEY": "${ANTHROPIC_API_KEY}"
        },
        "permissionMode": "approve-reads"
      },
      "gemini": {
        "command": "gemini",
        "args": ["--acp"],
        "env": {
          "GOOGLE_API_KEY": "${GOOGLE_API_KEY}"
        },
        "permissionMode": "deny-all"
      }
    }
  }
}

5.2 Binding 配置

可以将特定的 Discord 频道或 Telegram 群组绑定到 ACP，让用户在这些地方直接使用编程 Agent：

{
  "bindings": [
    {
      "agent": "main",
      "match": { "channel": "discord" },
      "acp": {
        "defaultAgent": "codex",
        "allowedAgents": ["codex", "claude-code", "gemini"],
        "autoThread": true
      }
    }
  ]
}
🔒 安全配置：allowedAgents 定义了该绑定中用户可以 spawn 的编程 Agent 白名单。不在列表中的 Agent 即使已配置也无法被 spawn。

6. 操作命令一览

命令语法说明
/acp spawn/acp spawn [agent] "task"启动一个新的 ACP 会话
/acp steer/acp steer "instruction"向运行中的 ACP 会话发送指令
/acp status/acp status查看所有活跃的 ACP 会话
/acp cancel/acp cancel [sessionId]取消正在执行的任务
/acp close/acp close [sessionId]关闭并清理 ACP 会话
/acp doctor/acp doctor诊断 ACP 环境和连接状态

6.1 spawn 命令详解

# 使用默认编程 Agent
/acp spawn "Add input validation to the API"

# 指定编程 Agent
/acp spawn codex "Fix the login bug"

# Session 模式
/acp spawn claude-code --mode session "Refactor the database layer"

# 指定仓库
/acp spawn codex --repo /path/to/project "Add error handling"

# 恢复已有会话
/acp spawn codex --resume abc123 "Continue where we left off"

6.2 steer 命令详解

# 在 Thread 中使用时（自动路由到绑定的 ACP Session）
"also add unit tests for the new validation"

# 显式指定会话
/acp steer --session abc123 "don't forget error messages"

# 纠正方向
/acp steer "stop, use PostgreSQL instead of MySQL"

7. 会话恢复（Resume Session）

ACP 支持跨设备、跨时间恢复编程会话。这对于以下场景特别有用：

  • 在手机上 spawn 了一个编程任务，回家后想在电脑上继续
  • 长时间运行的任务中断后恢复
  • 第二天继续昨天未完成的重构工作
# 1. 在手机上启动任务
/acp spawn claude-code --mode session "Refactor payment module"

# 查看会话 ID
/acp status
# → Session: abc123, Agent: claude-code, Status: active

# 2. 回家后在电脑上恢复
/acp spawn claude-code --resume abc123 "Continue the refactoring"
🔄 Resume 的工作原理：Gateway 会将 resumeSessionId 传递给编程 Agent。编程 Agent 会加载之前的会话上下文（代码变更历史、之前的指令等），无缝继续工作。不同编程 Agent 的 resume 能力取决于其自身实现。

8. 权限管理：permissionMode

每个 ACP Agent 可以配置权限模式，控制编程 Agent 能否直接执行操作：

模式行为适用场景
approve-all所有操作自动批准完全信任的环境（个人项目）
approve-reads读操作自动批准，写操作需确认大多数场景的推荐配置
deny-all所有操作需手动批准高安全要求的生产环境

8.1 权限模式配置示例

{
  "acp": {
    "agents": {
      "codex": {
        "command": "codex",
        "args": ["acp"],
        "permissionMode": "approve-all"
      },
      "claude-code": {
        "command": "claude",
        "args": ["--acp"],
        "permissionMode": "approve-reads"
      }
    }
  }
}

9. ACP vs Sub-agent 对比

ACP 和 Sub-agent 都能让主 Agent 派遣任务，但它们有本质区别：

维度ACPSub-agent
运行位置外部独立进程OpenClaw 进程内
运行时Codex/Claude Code 等专用运行时OpenClaw Agent Runtime
代码环境独立沙箱/容器，完整的开发工具链共享 Workspace，依赖宿主机工具
持久化ACP 会话（可跨设备 resume）Session Store（同一 Gateway 内）
交互方式/acp 命令 + Thread 绑定Agent 自动 spawn，结果自动 announce
编程能力专业编程 Agent，内置代码理解通用 LLM + 工具调用
配置复杂度需要安装和配置外部编程 Agent开箱即用，无需额外安装
成本取决于编程 Agent 的定价取决于 Sub-agent 的模型定价

9.1 何时用 ACP，何时用 Sub-agent？

场景推荐原因
写新功能代码ACP (Codex/Claude Code)专业编程 Agent 代码质量更高
修 bug / 跑测试ACP需要完整的代码执行环境
搜索信息 / 生成报告Sub-agent不需要代码环境，并行执行更高效
代码审查ACP编程 Agent 理解代码上下文更深
多源信息汇总Sub-agent并行搜索多个源，自动 announce
跨设备继续编程ACPresumeSessionId 跨设备恢复

10. 典型工作流示例

场景：在 Discord 中修复一个 Bug

# 1. 在 Discord #coding 频道发起任务

# 2. Gateway 自动创建 Thread "login bug fix" 并 spawn Codex

# 6. 完成后关闭
用户: /acp close
# → ACP Session 关闭，Thread 可以归档

12. 总结

ACP 是 OpenClaw 连接专业编程 Agent 的桥梁。它的核心价值在于：

  1. 统一入口：从任何通讯渠道 spawn 和管理编程 Agent
  2. 线程绑定：Discord/Telegram Thread 自动路由到同一个 ACP 会话，自然交互
  3. 会话恢复：跨设备 resume 编程会话，手机发起、电脑继续
  4. 权限可控：per-agent permissionMode 精确控制安全边界
  5. 与 Sub-agent 互补：ACP 做专业编程，Sub-agent 做通用任务
