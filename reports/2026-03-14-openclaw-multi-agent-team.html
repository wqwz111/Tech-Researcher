Executive Summary

OpenClaw 的多 Agent 架构让你可以像组建真人团队一样组建 AI Agent 团队：每个 Agent 有独立的能力（模型）、独立的工作空间（Workspace）、独立的权限（工具集），通过确定性的 Binding 路由系统分配任务。更进一步，Sub-agent 系统支持 Agent 动态派遣子任务给其他 Agent 并行执行，实现编排器（Orchestrator）模式。本文通过四个真实场景案例，从配置到部署完整展示多 Agent 协作的实现方式。

1. 多 Agent 架构基础

在单 Agent 模式下，所有消息都由同一个 AI 处理——相当于一个"万金油"员工。但现实中，不同场景需要不同能力的助手：

  • 日常闲聊 → 用便宜快速的模型（Gemini Flash）
  • 深度技术分析 → 用强大但贵的模型（Claude Opus）
  • 家庭群组 → 只能查天气和日历，不能执行代码
  • 代码开发 → 需要完整工具集 + 沙箱隔离

多 Agent 模式的本质是能力分层：把合适的能力（模型 + 工具 + Workspace）分配给合适的场景。

1.1 每个 Agent 的独立性

资源是否独立说明
Workspace✅ 独立每个 Agent 有自己的文件系统沙箱
AgentDir✅ 独立SOUL.md、AGENTS.md 等人设文件
Session Store✅ 独立会话历史不交叉
模型配置✅ 独立每个 Agent 可以用不同模型
工具集✅ 独立per-agent tools allow/deny
沙箱✅ 独立per-agent 沙箱容器
Gateway 进程❌ 共享所有 Agent 共享一个 Gateway

3. 任务路由：Binding 系统详解

Binding 系统是多 Agent 架构的"调度中心"。它定义了哪些消息应该路由到哪个 Agent。路由是确定性的——基于精确匹配规则，不是 LLM 判断。

3.1 匹配维度一览

维度匹配字段典型用途
渠道级别channelWhatsApp → Agent A, Telegram → Agent B
账号级别account多个 WhatsApp 号码分别路由
对话级别peer特定用户的 DM → 专用 Agent
群组级别guild / group特定 Discord 服务器 / WhatsApp 群组
角色级别roleDiscord 角色 → 不同 Agent
线程级别threadDiscord/Telegram 线程 → 特定 Agent

4. 实际案例

案例 A：按渠道分流

场景：WhatsApp 用于日常快速问答，Telegram 用于深度技术工作。日常用便宜快速的 Gemini Flash，深度工作用 Claude Opus。

{
  "agents": {
    "list": [
      {
        "id": "casual",
        "model": "openrouter/google/gemini-2.5-flash",
        "workspace": "./workspaces/casual"
      },
      {
        "id": "pro",
        "model": "openrouter/anthropic/claude-opus-4",
        "workspace": "./workspaces/pro"
      }
    ]
  },
  "bindings": [
    { "agent": "casual", "match": { "channel": "whatsapp" } },
    { "agent": "pro",    "match": { "channel": "telegram" } }
  ]
}

效果：WhatsApp 上的所有消息由 Gemini Flash 快速响应（成本低、延迟小），Telegram 上的消息由 Claude Opus 深度处理（质量高、能力强）。两个 Agent 的会话历史完全隔离，互不干扰。

💰 成本分析：Gemini 2.5 Flash 约 $0.15/1M input tokens，Claude Opus 4 约 $15/1M input tokens，价差 100 倍。按渠道分流可以显著降低日常使用成本。

案例 B：同一渠道内按对话分流

场景：你和老板都在 Telegram 上跟你的 AI 助手聊天。你希望给老板的消息由最强大的模型处理，自己的日常消息用便宜模型。

{
  "agents": {
    "list": [
      {
        "id": "vip",
        "model": "openrouter/anthropic/claude-opus-4",
        "workspace": "./workspaces/vip"
      },
      {
        "id": "personal",
        "model": "openrouter/google/gemini-2.5-flash",
        "workspace": "./workspaces/personal"
      }
    ]
  },
  "bindings": [
    {
      "agent": "vip",
      "match": {
        "channel": "telegram",
        "peer": "telegram:987654321"
      }
    },
    {
      "agent": "personal",
      "match": { "channel": "telegram" }
    }
  ]
}

注意：更具体的规则（指定 peer）放在前面，通用规则（仅匹配 channel）放在后面。这是 Binding 系统的首个匹配胜出机制决定的。

案例 C：家庭 Agent 绑定 WhatsApp 群组

场景：家庭 WhatsApp 群组中部署一个安全的 AI 助手，只能查天气和日历，不能读写文件或执行代码。

{
  "agents": {
    "list": [
      {
        "id": "family",
        "model": "openrouter/google/gemini-2.5-flash",
        "workspace": "./workspaces/family",
        "agentDir": "./agents/family",
        "tools": {
          "allow": ["read", "web_search", "weather"]
        },
        "sandbox": {
          "mode": "docker",
          "scope": "per-agent",
          "readonly": true
        }
      }
    ]
  },
  "bindings": [
    {
      "agent": "family",
      "match": {
        "channel": "whatsapp",
        "group": "whatsapp:family-group-id"
      }
    }
  ]
}

安全设计：

  • tools.allow 白名单：只允许 read、web_search、weather 三个工具
  • 没有 exec、write、edit → 无法执行命令或修改文件
  • Docker 沙箱 + readonly → 即使被 prompt injection 也无法破坏宿主机
  • 便宜模型 → 即使被滥用，成本可控

5. Sub-agent 系统

Sub-agent 是 OpenClaw 多 Agent 架构中最具威力的特性之一。它允许一个 Agent（主 Agent）在处理消息时，动态派遣子任务给其他 Agent 并行执行。

5.1 工作原理

用户消息 → 主 Agent 收到
         → 主 Agent 分析，发现需要并行执行多个任务
         → spawn Sub-agent A（搜索最新论文）
         → spawn Sub-agent B（分析代码库结构）
         → spawn Sub-agent C（生成信息图）
         → 三个 Sub-agent 并行工作...
         → Sub-agent A 完成 → 结果自动 announce 回主对话
         → Sub-agent B 完成 → 结果自动 announce 回主对话
         → Sub-agent C 完成 → 结果自动 announce 回主对话
         → 主 Agent 汇总所有结果，回复用户
🔄 Push-based 完成机制：Sub-agent 完成后，结果会自动推送回主 Agent 的会话（announce）。主 Agent 不需要轮询（poll）Sub-agent 的状态。这大大简化了编排逻辑，也避免了"忙碌等待"造成的资源浪费。

5.2 Sub-agent 配置

{
  "agents": {
    "defaults": {
      "model": "openrouter/anthropic/claude-sonnet-4",
      "subagents": {
        "model": "openrouter/google/gemini-2.5-flash",
        "maxConcurrent": 5,
        "runTimeoutSeconds": 300,
        "maxSpawnDepth": 1
      }
    }
  }
}
参数说明推荐值
modelSub-agent 使用的模型便宜快速的模型（如 Gemini Flash）
maxConcurrent同时运行的最大 Sub-agent 数量3-5
runTimeoutSecondsSub-agent 最大运行时间（秒）300（5 分钟）
maxSpawnDepth最大嵌套深度1（禁止嵌套）或 2（允许一层嵌套）

5.3 Sub-agent 的典型用途

  • 并行搜索：同时搜索多个信息源，汇总结果
  • 并行分析：同时分析多个文件/模块
  • 后台任务：长时间运行的任务（如生成报告）不阻塞主对话
  • 成本优化：主 Agent 用高质量模型做决策，Sub-agent 用便宜模型做执行

6. 嵌套 Sub-agent：编排器模式

当 maxSpawnDepth=2 时，Sub-agent 本身也可以派遣 Sub-agent，形成两层结构：

主 Agent（编排器）
  ├── Sub-agent Layer 1（协调者）
  │     ├── Sub-agent Layer 2（工作者 A）
  │     ├── Sub-agent Layer 2（工作者 B）
  │     └── Sub-agent Layer 2（工作者 C）
  └── Sub-agent Layer 1（另一个协调者）
        ├── Sub-agent Layer 2（工作者 D）
        └── Sub-agent Layer 2（工作者 E）

6.1 编排器模式的工作原理

在编排器模式下，三层 Agent 各司其职：

  1. 主 Agent：理解用户意图，决定整体策略，不执行具体任务
  2. Layer 1 协调者：接收主 Agent 的子任务，分解为更细粒度的任务，派遣给 Layer 2 工作者
  3. Layer 2 工作者：执行具体的搜索、分析、生成任务，将结果返回给协调者

6.2 工具策略

在嵌套 Sub-agent 场景中，工具权限可以逐层递减：

{
  "agents": {
    "defaults": {
      "model": "openrouter/anthropic/claude-sonnet-4",
      "tools": {
        "allow": ["read", "write", "edit", "exec", "web_search", "process"]
      },
      "subagents": {
        "model": "openrouter/google/gemini-2.5-flash",
        "tools": {
          "allow": ["read", "web_search", "write"]
        },
        "maxSpawnDepth": 2
      }
    }
  }
}

主 Agent 有完整的工具集（包括 exec），但 Sub-agent 只有 read、web_search 和 write，不能执行 shell 命令。这种权限递减策略是企业安全的重要保障。

6.3 级联停止机制

当主 Agent 的会话结束或超时时，所有嵌套的 Sub-agent 会被级联停止：

  • 主 Agent 会话结束 → 所有 Layer 1 Sub-agent 收到停止信号
  • Layer 1 Sub-agent 停止 → 所有 Layer 2 Sub-agent 收到停止信号
  • 整个树状结构被干净地终止，不会留下孤立进程
⚠️ 编排器模式的注意事项：
  • 每增加一层嵌套，延迟和成本都会增加
  • Layer 2 工作者无法直接与用户交互，所有结果需要逐层传递
  • 建议从 maxSpawnDepth=1 开始，只在真正需要时升级到 2
  • 设置合理的 runTimeoutSeconds 防止嵌套 Sub-agent 失控

8. 总结与选择建议

场景推荐模式模型选择
个人日常使用单 AgentClaude Sonnet / Gemini Flash
多渠道、不同场景多 Agent + Binding按渠道差异化
团队共享 Gateway多 Agent + account 绑定每人独立选择
复杂任务并行主 Agent + Sub-agent主用强模型，Sub 用便宜模型
企业级编排编排器模式 (depth=2)分层差异化
